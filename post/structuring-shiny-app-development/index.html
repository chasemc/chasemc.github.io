Structuring Shiny App Development

Dec 7, 2018 5 min read Instructional, R, Shiny

Part One: Scope

This is going to be a fairly opinionated post about my not novel but recent revelations as to how I think Shiny apps should be formatted. By formatting I don’t mean UI formatting, code style, etc. I mean how one might organize, holistically, a Shiny app to help maximize readability, stability, and scalability. Again not novel, just me getting my thoughts “on paper’.

Before now my server() sections kind of looked like this:

I am aware that it is possible to “modularize” a shiny app and there is good info on that from RStudio here and Steph Locke here.

I am not a fan of this approach for a couple of reasons:

  • I think it can add unnecessary complexity to small applications
  • I think it’s difficult to use with large applications that have highly interconnected reactivity
  • But, in reality, it’s likely I just haven’t spent enough time trying it ¯\(ツ)/¯

Part Two: Straight to the Point

I think abstracting Shiny apps to small functional components is most helpful when done at the most fundamental level. Luckily there are two elements of the R ecosystem that are crazy-complementary to Shiny apps:

  • Packages
  • Functions

Those links point to Hadley Wickham’s book R packages and a chapter of Advanced R, respectively.

And if we look at the first sentence of each:

R packages:
“In R, the fundamental unit of shareable code is the package.”

Advanced R:
“Functions are a fundamental building block of R…”

i.e. The tools you need to organize your app exist:

  • Start by making it a package
  • Then break your computation into standalone functions that reside outside the app

Part Three: A Shiny App is a Package

Making your Shiny app a package allows you to:

  • Easily distribute the app
  • Manage package dependencies
  • Write unit tests for your app (more on that in Part Five)
  • Easily include any needed data within the app (eg example data, models)
  • Organize your code

Creating an app as a package provides you a framework for development:

Dean Attali briefly described the basics of including a Shiny app within a package here. That formed the basis for how I now write Shiny apps, along with what I’ve learned from creating Shiny apps as Windows executables using Jonathan Hill’s “RInno” package

Part Four: Shiny Apps Consist of Functions

Concept One: Reactive Expressions are Basically Functions

In reality it’s more complicated but a lot of the basic components of a Shiny app (e.g. server(), reactive(), observe(), etc.) can be thought of as functions.

Imagine we have a simple app that has a slider input that represnents the number of pumpkins you sell per year, but you want the log10 of that number.

You could make a little reactive expression that would do this:

pumpkinLog <- reactive({
                log10(input$pumpkins)
              })

Now imagine we moved the slider input to 10 pumpkins. We then save pumpkinLog() to savedValue in the parent environment and close the app.

observe(
 savedValue <<- pumpkinLog()
)

What will printing savedValue give us?

print(savedValue)
   
  [1] 1

It gave us 1, the value we would expect to be returned from log10(10).

This is because, in essence, you are asking for the result of calling a reactive “function”.

And, just like in base R, functions can take as arguments the result given by calling another function.

Concept Two: Shiny Inputs Just Return a Value

This concept is pretty simple, input$... just stores the value of inputs. That’s phenomenally important because it means we can use input$... as an argument for another function.

If you don’t believe me check out below where I saved the inputs from an intermediate point within an app. And upon closing the app printed the values.

myInputs <<- as.list(input)
# close app

print(myInputs)

Part Five: Putting it All Together

So now we know we can use the value from calling a reactive(...) expression and the value returned by an input$... as inputs to functions.

That means we can do this:

# Saved as a seperate .r file, within R folder of package directory
# eg "pumpkinLogarithm.r"" would contain this:

pumpkinLogarithm <- function(pumpkins){
   log10(pumpkins)
}     

# Saved as a seperate .r file, within R folder of package directory
# eg "otherProduce.r""  would contain this:

totalProduce <- function(tomatoes, apples, loggedPumpkin){
  log10(tomatoes) + log10(apples) + loggedPumpkin
}

# Within your app's server function

pumpkinLog <- reactive({
                pumpkinLogarithm(pumpkins = input$numberOfPumpkins)
              })

groceries <- reactive({
               totalProduce(tomatoes = input$tomatoes,
                            apples   = input$apples,
                            pumpkins = pumpkinLog()
                            )
             })

So now you have two functions that live outside the Shiny app. Your server file is more succinct, especially in regard to the variables within each reactive or observe chunk that may have had reactive side effects.

Most importantly:

  • Because you are writing package functions you can wield the freaking awesome power of testthat!

I realize that this only tests the internals/non-reactiveness of an app, but there are other packages to test layout/reactivity, etc. But these can be hard to keep up with large apps that don’t have a stable codebase. For that you can reference the packages below
