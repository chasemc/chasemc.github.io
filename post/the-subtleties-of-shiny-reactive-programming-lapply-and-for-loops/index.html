The Subtleties of Shiny Reactive Programming: Lapply and For Loops

Sep 27, 2019 3 min read Shiny

Why Read This?

  1. You want to understand why for loops can fail to produce desired results in an R Shiny app.
  2. You want an example of creating a dynamic number of plots within an R Shiny app.

A Minimal, Reproducible Example

First, letâ€™s look at the minimal reprex. This is an R Shiny app that uses modules to display a dynamic number of plots.

  • plot_overlay_ui() is a Shiny module ui function that returns a <tag> list containing our plot(s).

  • plot_overlay_server() is a Shiny module server function that creates the plot.

  • ui() is a regular Shiny ui function that contains the code to create the main skeleton of the user interface.

  • server() is a regular Shiny server function that contains logic to run the app.

  • plot_data() is a reactive expression that returns a list, where each list element is a vector representing a column of mtcars selected using the checkboxGroupInput.

  • multi_plot_ui renders ui/html that is dynamically generatedâ€¦ this will hold our dynamically-generated plots

  • And callModule(plot_overlay_server,...) is where the focus of this blog takes place.

library(shiny)

plot_overlay_ui <- function(id) {
  ns <- NS(id)
  tagList(
    plotOutput(ns("my_plot"), 
               width = "100%")
  )
}
# -------------------------------------------------------------------------

plot_overlay_server <- function(input,
                                output,
                                session, 
                                spec,
                                plot_name) {
  
  output$my_plot <- renderPlot({
    
    plot(spec, 
         main = plot_name)
    
  })
}
# -------------------------------------------------------------------------

ui <- fluidPage(
  titlePanel("Reprex"),
  sidebarLayout(
    sidebarPanel(
      checkboxGroupInput("selected_sample",
                         label = "Select Column",
                         choices = seq_along(mtcars))
    ),
    mainPanel(
      uiOutput("multi_plot_ui")
    )
  )
)
# -------------------------------------------------------------------------

server <- function(input, output) {
  
  plot_data <- reactive({
    col_select <- as.integer(input$selected_sample)
    lapply(col_select, function(x) mtcars[,x])
  })
  
  observeEvent(input$selected_sample, {
    req(!is.null(input$selected_sample))
    req(plot_data())
    
    output$multi_plot_ui <- renderUI({
      
      lapply(seq_along(input$selected_sample),
             function(n) {
               return(plot_overlay_ui(paste0("n", n)))
             })
    })
    
    lapply(seq_along(input$selected_sample),
           function(i){
             callModule(plot_overlay_server,
                        paste0("n", i),
                        spec = plot_data()[[i]],
                        plot_name = i)
           }
    )
    
    # for (i in seq_along(input$selected_sample)) {
    #   callModule(plot_overlay_server,
    #              paste0("n", i),
    #              spec = plot_data()[[i]],
    #              plot_name = i)
    # }
    
  })
}

shinyApp(ui = ui, server = server)

Lapply vs For Loops in Shiny

My original code to generate the multiple server functions to generate plots was:

# the for loop
for (i in seq_along(input$selected_sample)) {
  callModule(plot_overlay_server,
             paste0("n", i),
             spec = plot_data()[[i]],
             plot_name = i)
}

While this was able to produce multiple plots, all of the plots were identical and represented the plot created during the last iteration of the loop. I was convinced the problem was a regression in my Shiny abilities and that I was simply missing an session$ns() or the like.

After convincing myself everything was correct, I tried an experiment that I thought was crazy enough to work because of the underlying mechanicsâ€¦ change the for-loop to a lapply.

# the lapply
lapply(seq_along(input$selected_sample),
       function(i){
         callModule(plot_overlay_server,
                    paste0("n", i),
                    spec = plot_data()[[i]],
                    plot_name = i)
       }
)

And the lapply worked! Multiple plots representing the correct, selected data were displayed!

My best guess was that for a lapply() every iteration was run in its own environment. This turns out to be exactly what allows this to work, and I am honestly surprised I hadnâ€™t encountered this before.
